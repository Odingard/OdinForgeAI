# OdinForge v1.1 — Parallel Architecture + LLM Boundary Enforcement

## Overview

v1.1 introduces three capabilities governed by two specification documents:

1. **Parallel Specialized Agents** — MicroAgentOrchestrator replaces sequential Phase 1B with massively concurrent endpoint × vulnerability class fan-out
2. **Runtime Context Broker** — Go agent telemetry injected into LLM prompts for Cloud IAM and Lateral Movement phases
3. **Auto-Remediation Loop** — Fix proposal generation + verification for proven vulnerabilities

The **LLM Boundary Amendment** is the governance layer enforced across all three capabilities, structurally preventing LLM-generated findings from reaching customer reports.

## Capability 1: Parallel Micro-Agent Orchestrator

### Architecture

```
MicroAgentOrchestrator
  ├── buildAgentSpecs(endpoints, scope, chainId, targetUrl) → MicroAgentSpec[]
  │     └── isVulnClassApplicable() filters ~30-40% unnecessary agents
  ├── dispatch(specs, onProgress) → MicroAgentResult[]
  │     └── Semaphore-gated: up to 50 concurrent agents
  │     └── Token-bucket rate limiter: 50 req/sec per target hostname
  └── runMicroAgent(spec) → MicroAgentResult
        ├── generatePayloads() — deterministic, no LLM
        ├── firePayload() — real HTTP via fetch(), returns RealHttpEvidence
        ├── classifyEvidence() — deterministic regex pattern matching
        └── buildFinding() — throws if confirmedEvidence.length === 0
```

### Key Files

| File | Purpose |
|------|---------|
| `server/services/aev/micro-agent-orchestrator.ts` | Core fan-out engine |
| `server/services/agent-rate-limiter.ts` | Token-bucket rate limiter |
| `server/lib/semaphore.ts` | Counting semaphore for concurrency control |
| `server/lib/real-evidence.ts` | `RealHttpEvidence` type + runtime constructor |

### Vulnerability Classes

10 classes with specific payloads and detection patterns:

| Class | Payloads | Detection |
|-------|----------|-----------|
| sqli | `' OR 1=1--`, `1 UNION SELECT`, `' AND SLEEP(5)--` | SQL error strings, timing > 4s |
| xss | `<script>alert(1)</script>`, `"><img src=x onerror=alert(1)>` | Reflected payload in response |
| ssti | `{{7*7}}`, `${7*7}`, `<%= 7*7 %>` | `49` in response body |
| cmdi | `; id`, `| whoami`, `` `id` `` | `uid=`, `root:` in response |
| path_traversal | `../../../etc/passwd`, `..\\..\\..\\windows\\win.ini` | `root:x:`, `[extensions]` |
| ssrf | `http://169.254.169.254/latest/meta-data/` | Cloud metadata in response |
| auth_bypass | `admin' --`, direct admin URL access | Admin content without auth |
| idor | Sequential ID enumeration | Different user data returned |
| header_injection | `\r\nX-Injected: true` | Injected header in response |
| open_redirect | `//evil.com`, `/\evil.com` | Location header to external domain |

### Applicability Filter

Reduces unnecessary agent spawns by ~30-40%:

- Skip `sqli` on endpoints with no query/body parameters
- Skip `ssrf` on endpoints without URL-type parameters
- Skip `auth_bypass` on already-unauthenticated endpoints
- Skip `open_redirect` on non-redirect endpoints

### Integration with Breach Orchestrator

Phase 1B-micro runs between Phase 1A (Active Exploit Engine) and Phase 1B (AI Pipeline):

```typescript
// If micro-agents find ≥3 findings, skip the AI pipeline (fallback only)
if (microAgentFindings >= 3) {
  // AI pipeline skipped — micro-agents provided sufficient coverage
}
```

WebSocket event `breach_chain_agent_dispatch` streams per-agent progress to the frontend.

## Capability 2: Runtime Context Broker

### Architecture

```
RuntimeContextBroker
  ├── getContextForAsset(assetId) → HostRuntimeContext | null
  │     ├── storage.getAgentByAssetId() — cloudAssets → discoveredAssets fallback
  │     └── storage.getLatestAgentTelemetry()
  ├── deriveSignals(telemetry) → RuntimeSignal[]  ← DETERMINISTIC, ZERO LLM
  │     ├── Port-based: DB_PORTS, WEB_PORTS, SSH
  │     ├── Service-based: K8S, MQ, cache, container, monitoring
  │     ├── Security findings from agent
  │     └── Config file signals (env_file, cloud_credential_file)
  └── formatForPrompt(ctx) → string
        └── Includes boundary: "Reason ONLY from the data above"
```

### Signal Types

| Signal Type | Severity | Source |
|---|---|---|
| `database_service` | high | Ports 3306, 5432, 27017, etc. or service name match |
| `web_server` | info | Ports 80, 443, 8080, etc. |
| `container_runtime` | high | docker, containerd, podman detected |
| `kubernetes` | critical | kubelet, kube-apiserver, etcd detected |
| `message_queue` | medium | rabbitmq, kafka, nats detected |
| `ssh_service` | medium | Port 22 open |
| `env_file` | high | `.env` file with credential keys detected |
| `cloud_credential_file` | critical | `.aws/credentials`, `.kube/config` detected |
| `security_finding` | varies | Agent-reported security issue |

### Go Agent Config Files Collector

`odinforge-agent/internal/collector/config_files.go`:

- Scans well-known paths: `.env`, `.aws/credentials`, `.kube/config`, `config/database.yml`
- Matches credential key patterns: `PASSWORD`, `API_KEY`, `ACCESS_KEY`, `TOKEN`, `DATABASE_URL`
- **Values are NEVER read or transmitted** — only key names are reported
- Results sent as `configFiles` in telemetry payload, mapped to `configData` on server

### Injection Points

- **Phase 3 (Cloud IAM Escalation):** Runtime context appended to `contextDescription` for each asset
- **Phase 5 (Lateral Movement):** Runtime contexts fetched for all compromised assets, passed to `LateralMovementSubAgent`

## Capability 3: Auto-Remediation Loop

### Architecture

```
generateFixProposal(finding, chainId)
  ├── Evidence quality gate: PROVEN/CORROBORATED only (throws for INFERRED)
  ├── inferFixType() — deterministic: IAM, WAF, network, config, code
  ├── buildFixContent() — pattern-matched fix templates
  └── buildVerificationPayload() — original exploit payload + vuln indicators

verifyFix(proposal)
  ├── Re-fire original payload via fetch() — ZERO LLM
  ├── Check if vulnerability indicators are gone
  └── Return: confirmed_fixed | still_vulnerable | partially_fixed | error
```

### Fix Types

| Type | When Generated | Example |
|---|---|---|
| `waf_rule` | XSS, SQLi, injection findings | ModSecurity `SecRule` directives |
| `iam_policy` | IAM escalation findings | AWS IAM deny policy for dangerous actions |
| `network_control` | Open port, firewall findings | iptables rules to restrict access |
| `config_change` | Misconfiguration findings | Secure configuration recommendations |
| `code_fix` | Default for all other findings | Code-level remediation guidance |

### API Endpoints

| Method | Endpoint | Purpose |
|---|---|---|
| `POST` | `/api/breach-chains/:id/generate-fix` | Generate fix proposals for all eligible findings |
| `GET` | `/api/breach-chains/:id/fix-proposals` | List fix proposals for a chain |
| `POST` | `/api/breach-chains/:id/verify-fix-proposal` | Re-fire exploit to verify fix |

### Background Generation

After breach chain completion, `generateFixProposalsForChain()` runs fire-and-forget:
- Only generates for PROVEN/CORROBORATED findings
- Only critical/high severity
- Capped at 20 proposals per chain

## LLM Boundary Enforcement

### Contract: Where LLM is Allowed vs Forbidden

**Allowed:**
- Classify real HTTP response bodies
- Vary payload selection based on observed behavior
- Narrate findings from real evidence
- Suggest remediation for proven vulnerabilities
- Reason about attack chains from real data

**Forbidden:**
- Generate findings without real HTTP evidence
- Estimate vulnerabilities or produce synthetic responses
- Short-circuit Phase 1A (Active Exploit Engine)
- Populate `timeToCompromise` with estimates
- Fill missing evidence fields with fabricated data

### Structural Enforcement

| Layer | Mechanism | File |
|---|---|---|
| Type system | `RealHttpEvidence` with runtime validation | `server/lib/real-evidence.ts` |
| Agent execution | `firePayload()` — zero LLM, pure `fetch()` | `micro-agent-orchestrator.ts` |
| Finding creation | `buildFinding()` throws on empty evidence | `micro-agent-orchestrator.ts` |
| Result merging | `mergeMicroResults()` discards empty-evidence findings | `micro-agent-orchestrator.ts` |
| Quality gating | No source + no real evidence → UNVERIFIABLE | `evidence-quality-gate.ts` |
| Report output | INFERRED/UNVERIFIABLE suppressed from customer reports | `report-generator.ts` |
| Remediation | `generateFixProposal()` rejects non-PROVEN/CORROBORATED | `remediation-engine.ts` |
| CI | Grep guard: zero LLM imports in active-exploit-engine.ts | `scripts/ci/check-phase1a-no-llm.sh` |

### Evidence Quality Tiers

| Tier | Classification | Passes Gate | In Report |
|---|---|---|---|
| PROVEN | Real HTTP response with statusCode + body | Yes | Yes |
| CORROBORATED | Active exploit engine source or protocol auth | Yes | Yes |
| INFERRED | LLM inference or heuristic pattern match | No | No (suppressed) |
| UNVERIFIABLE | No source, no real evidence | No | No (suppressed) |

## Deployment Notes

### Database Migration

New columns added to `breach_chains` table:
- `fix_proposals` (jsonb, default `[]`)
- `fix_verifications` (jsonb, default `[]`)

These use Drizzle defaults and will be created automatically on first access. No manual migration required.

### New Environment Variables

None required. All new features use existing configuration.

### Breaking Changes

- `separateByQuality()` in report-generator.ts returns `suppressedFindings` instead of `additionalObservations`
- `parallel-agent-dispatcher.ts` default `maxConcurrentAgents` changed from 5 to 10

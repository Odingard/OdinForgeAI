# OdinForge AEV v1.1 — Modular Agent Architecture Roadmap

> Post-launch architectural improvement. Does NOT block GTM v1.0 launch.
> Status: **Planned** | Target: v1.1 (post first customer engagement)

## Motivation

The v1.0 breach orchestrator works — all 6 phases execute, credentials propagate,
evidence is real. But the architecture is monolithic: `breach-orchestrator.ts` is
2300+ lines with inline phase executors. v1.1 extracts each phase into a modular
agent class for testability, scalability, and team parallelism.

## Architecture Target

### Agent Interface

```typescript
interface Agent {
  phase: string;
  name: string;
  run(inputs: Finding[], ctx: PhaseContext): Promise<Finding[]>;
  healthCheck(): Promise<boolean>;
}
```

Each phase executor becomes a class implementing `Agent`. The orchestrator
becomes a thin loop that loads registered agents and chains `run()` results.

### Target Folder Structure

```
server/services/agents/
├── interface.ts              # Agent interface + PhaseContext types
├── registry.ts               # Agent registration + discovery
├── phases/
│   ├── application-compromise.agent.ts   # Phase 1 (wraps active-exploit-engine)
│   ├── credential-extraction.agent.ts    # Phase 2 (regex parser + LLM fallback)
│   ├── cloud-iam-escalation.agent.ts     # Phase 3 (AWS pentest service)
│   ├── k8s-breakout.agent.ts             # Phase 4 (K8s pentest service)
│   ├── lateral-movement.agent.ts         # Phase 5 (PivotQueue + sub-agents)
│   └── impact-assessment.agent.ts        # Phase 6 (aggregation)
├── workers/
│   ├── exploit-worker.ts     # Docker-isolated exploit execution
│   └── pivot-worker.ts       # Isolated lateral movement worker
└── queue/
    ├── bull-config.ts         # BullMQ + Redis configuration
    └── phase-queue.ts         # Per-phase job queues
```

## Improvements by Priority

### P0 — BullMQ Job Queues (replaces in-memory phase execution)
- **Why**: Current in-memory execution loses state on worker crash mid-engagement
- **What**: Each phase becomes a BullMQ job; Redis persists queue state
- **Benefit**: Resume mid-engagement after restart, concurrent engagement scaling
- **Effort**: 3-4 days
- **Dependencies**: Redis (already deployed in prod docker-compose)

### P1 — Agent Interface Extraction
- **Why**: breach-orchestrator.ts is 2300+ lines; testing requires mocking the entire file
- **What**: Extract each `executeXxx()` function into a class implementing `Agent`
- **Benefit**: Unit test each phase in isolation, swap implementations
- **Effort**: 2-3 days (mechanical refactor, no behavior change)
- **Dependencies**: None

### P2 — Docker Isolation for Exploit Workers
- **Why**: Active exploit payloads run in the main Node process
- **What**: Spawn exploit execution in ephemeral Docker containers with network isolation
- **Benefit**: Blast radius containment, resource limits, audit trail
- **Effort**: 3-4 days
- **Dependencies**: Docker socket access on host

### P3 — Prometheus Metrics + Structured Observability
- **Why**: No quantitative visibility into engagement performance
- **What**: Export metrics via `/metrics` endpoint
- **Metrics**:
  - `odinforge_engagement_duration_seconds` (histogram)
  - `odinforge_phase_success_total` (counter by phase)
  - `odinforge_credentials_harvested_total` (counter)
  - `odinforge_evidence_quality_ratio` (gauge: proven/total)
  - `odinforge_detection_rules_generated_total` (counter)
  - `odinforge_pivot_depth_max` (gauge)
- **Effort**: 1-2 days
- **Dependencies**: prom-client npm package

### P4 — CredentialBus → Redis Streams
- **Why**: Current in-memory pub/sub loses credential state on restart
- **What**: Replace EventEmitter-based bus with Redis Streams consumer groups
- **Benefit**: Durable credential propagation, multi-worker support
- **Effort**: 2 days
- **Dependencies**: Redis Streams (Redis 5+, already deployed)

### P5 — Configuration Externalization
- **Why**: Ports, regexes, timeouts hardcoded in source
- **What**: YAML/JSON manifests for phase config, payload lists, port maps
- **Benefit**: Update without rebuild, customer-specific tuning
- **Effort**: 2 days

## Migration Path

1. Extract Agent interface + registry (P1) — no behavior change
2. Wire BullMQ behind the existing orchestrator loop (P0) — drop-in replacement
3. Move exploit execution to Docker workers (P2)
4. Add Prometheus (P3) — additive, no breaking changes
5. Swap CredentialBus internals to Redis Streams (P4) — same API surface
6. Externalize config (P5)

Each step is independently deployable. No big-bang rewrite.

## What NOT to Change

- Phase execution order (hardcoded dependency chain is correct)
- Credential encryption (AES-256-GCM in credential-store.ts is production-grade)
- Evidence Quality Gate, Defender's Mirror, Replay Recorder, Reachability Chain (just shipped in v1.0)
- Active Exploit Engine internals (highest-value component, already real)
- Schema types in shared/schema.ts (stable contract between server + client)

## Estimated Timeline

Conservative: 3-4 weeks post-launch (P0-P3)
Full completion: 5-6 weeks (all P0-P5)
